Extending the autocomplete attribute

This is an effort to analyze the structures of address forms in a somewhat machine readable way.

This work is based on the following assumptions:

  • The autocomplete spec was written with the intent to find a lowest common denominator that should work for all countries. An analysis of real-world web-forms indicates that those forms cannot be properly annotated with today's autocomplete attribute because it lacks expressiveness. Probably many site owners don't have a choice because shipping companies or local customs tell them how an address form should look like.
  • There are certain aspects that just apply to a few countries (e.g. Japan has phonetic names, Spain has paternal and maternal last names, Hungary splits the name of a street from the type of a street). The fact that these concepts only apply to a few countries does not make them go away. We should allow storing a few extra fields for Japanese, Spanish, Hungarian, ... addresses (i.e. decide which fields to offer based on the country of an address) because oftentimes customers and merchants are in the same country and share the same cultural norms when it comes to address forms.
  • I don't think that we can immediately fix autofill for all countries and this analysis does not cover all countries. I hope that this document leads to an architecture that can be extended.
  • We may need to combine certain atomic pieces of information into a single field (e.g. a house number and an apartment number). This process is probably very locale specific and needs to grow over time. The fact that we don't know how to combine two fields into one fillable string for some countries yet, hopefully does not invalidate the overall idea.

Note that this is work in progress. Feedback is highly appreciated.

Big and global questions

  • Do we want to support multiple concurrent field types? For example a single <input> field for "name or company". I don't see an obvious syntax for adding this capability to the autocomplete attribute. If you search for '[' below, you can find examples where a website requested one (or more) of multiple field types. Several such cases are already expressed via new compound field types like 
    street-location
    (street name and house number) that I am proposing here.
  • How do we model the "everything else" field type? Do we want to call it "address-overflow", "address-supplement" (this seems to be a term that is quite common in many countries), "building-and-location-in-building" (also common)? The latter may be too narrow because it does not cover a company name or c/o. Some websites would not want a company or c/o in their "everything else" fields while others would want them.
    Instead of an "everything else" type we could ask websites to enumerate what should go into these fields (see the previous bullet point).
  • Are we ok with introducing country specific concepts? E.g. Hungary separates the name of a street (e.g. "Amphitheatre") from the type of the street ("Ave", "Pkwy", "Blvd", ...). I haven't seen this anywhere else. It's essential to fill Hungarian address forms but we should not offer this to all users.
    How far do we go? In Spain we see the "Tipo de vÃ­a" but websites offer different options to choose from. Do we want to prescribe a list of supported road types?
  • Do we allow forms to specify an encoding? E.g. Japanese forms are picky about ASCII, full-width or half-width characters.
  • The patronymic name is based on the first name of the father. As per Eastern Slavic naming customs, it is written in these countries in all legal and identity documents. Countries mentioned there include Russia, Belarus, Ukraine, Kazakhstan, Turkmenistan, Uzbekistan, and to an extent in Kyrgyzstan, Georgia. But the list of affected countries is longer.
  • Many naming questions...
    • What do we call the field type to find a place inside a building? "location-in-building"? "sub-building"?
    • What do we name things that are very country/culture specific (Hungary's separation between street name and street type, entrecalles, stroenie/korpus, ...)?
  • How do we deal with websites whose address forms that centered around a search widget? Do we want to offer autofilling?

# AR - Argentina ğŸ‡¦ğŸ‡·

Argentinian address forms rely on street names, house numbers, floors (Piso) and apartments (Departamento). Floors and apartments can be requested in a single input field or in two separate fields.

On the street address level, we frequently see between streets, i.e. a field in which one specifies the names of two crossing streets to help locate the house number.

The forms ask for states/provinces and localities/cities.

Oftentimes the address form was filled via an address search box.

Frequently a DNI was requested (national identification number).

Open questions

  • How do we deal with address-overflow? Oftentimes, this seems to ask for information within a building - floor and apartment - but theoretically it could be more flexible. Do we want to rely on "location-in-building"?
  • Do we want to support entrecalles (between streets) which are split across two fields? - Do we want to combine this with "landmark"?
  • Two forms asked for a "Lote" - a piece of land (a lot / parcel of land).
  • One form asked for a "Torre", which literally means a tower but in this context it would refer to a part of a building (e.g. front-building, a wing of a building, ...).
  • The concept of "partido" is not fully clear to me. One form asked for (provincia, partido, localidad), which I see as (province, part of a city, city). Another form asked for (partido, provincia) but no city. Maybe this was implied by the zip code?
  • Should we model the entrecalles ("between streets") as a landmark with a country specific translation (i.e. translate "landmark" as "entrecalles" in AR and MX)? Only one website in Mexico had separate fields for entrecalles and a landmarks. Several websites combined these concepts into a field "betreen streets or landmark". They serve the same purpose.

# IN - India ğŸ‡®ğŸ‡³

Indian addresses forms have surprising naming conventions. Forms commonly ask for a 'House no/Apartment name'. The 'house number' is an apartment number (not the sequence of a house in a street) and the apartment name is the name of a building (most buildings have a name; this is not referring to the name of an apartment).

Another important aspect that is not common in other countries is a 'locality' (sometimes referred to as 'area details'). An example would be 'HITEC City' in Hyderabad. It's a name for an area that people know and use for orientation. Another example would be 'old town'.

A 'Colony' is a named group of buildings that belong to each other, e.g. sharing amenities. Also serves as a landmark.
